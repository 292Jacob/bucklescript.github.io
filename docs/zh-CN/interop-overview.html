总览

Interop 是 interoperability （互操作）的缩写，在BuckleScript中，Interop表示与javascript进行通信。 BuckleScript的核心目标之一就是尽可能平滑的与跨堆栈javascript进行交互。

实际开发中，一个常规的代码库，很难在短短几天之内，从一种技术转换成另一种技术的实现，这个过程可能会花费几个月甚至几年。 对于一个互操作性没有充分仔细的被设计的语言， 如果还在犹豫是否应该引入到代码库中的时候，转换代价在初期将成为这个语言的价值的短板，并最终放弃这个语言。

一个精心设计的互操作性的语言应该是简单，高效，本地的和非指令性的，允许开发者使用熟悉的概念去使用这个语言，并且应该是足够自信的去推荐给对这个技术不是很了解的上级管理者，并且当用于团队的项目之后应该是高产的。

申明

上面的小节可能听起来有点buzzword-y，这也是为什么我们从栈的最底层到最上层的设计了BuckleScript的互操作。

变量

大部分变量的名称能被编译成了干净的js名称， hello编译成了hello，但有些可能会编译成比如__$INTERNAL$hello$1， 干净的名称（没有被混淆过的）能够更好的调试。 我们见过一个团队中，有对BuckleScript和Reason不是那么熟悉的成员需要紧急修复bug的场景，直接在BuckScript文件中写一段原始的js代码。 邻人欣慰的是BuckleScript设计了这样的"最后逃生窗口"，实在没招的情况下这种方式也是支持的。

数据结构

BuckleScript（简称BS）的主要数据结构完全映射到了JS的数据结构，例如，BS 中的string 就是js中的string，BS 中的 array 就是 js中的 array，下面就是BS的代码：

let messages = [| "hello"; "world"; "how"; "are"; "you" |]

Reason 语法:

let messages = [|"hello", "world", "how", "are", "you"|];

编译后的js代码

var messages = ["hello", "world", "how", "are", "you"]

在使用字符和数组时，不需要有心理的担忧，是不会有性能上的损耗的，在BuckleScript这一侧的代码将自动转成为相应的数组之类的东西。

但并不是所有的复杂数据结构都是这样的，可以参考具体的章节查看更多的内容。

函数

绝大多数情况下，可以在BS中直接调用JS的函数，反之亦然，在大多数情况下，这些调用也是没有性能开销的。

模块/文件

每一个在BS中定义的let都会默认被导出，并且在js中使用， 同样的，你也可以在BS文件中定义你想要的调用JS模块， 可以最终输出CommonJS，ES6或者AMD模块规范

输出的代码是非常干净的，几乎接近手写的javascript代码，你可以在playground中试一试。

构建系统

BS配备了一个闪电般快速的构建系统， 它在几毫秒内就能启动并很快能完成， 增量编译也能在两位数毫秒内完成。 这使得构建系统可以无形地插入到你的整个JS构建管道线中而不会尴尬， 除非你的JS构建管道已经非常慢了。 不过那也没关系。

一个BS文件编译成一个JS文件， 可以配置成为让生成的JS文件在ml/re源代码内或者外部。 这意味着你都不必寻求团队的允许就可以在公司里尝试BuckleScript，可以轻松的生成JS文件并提交。 从后面的编译管道线来说，就像是你手写了这些JS代码。 这就是为什么Messenger successfully introduced Reason into the codebase。

包管理

我们使用NPM和Yarn。 因为生成的代码足够的干净，你可以在npm上发布，只要你事先删除BuckleScript的痕迹。 用户基本上不会感知这个库使用BS写的， 就算看node_moudules，可能也只能看到一些看不出来是BS写的，转换过的BS代码。

结论

希望你能看到上面这些概述，因为我们在互操作性的架构里为了简洁、一致性和性能倾注了大量的思考和精力。 接下来的章节将回详细介绍这些要点。
