Configuración

bsconfig.json es el único y obligatorio meta archivo de construcción necesario para bsb.

El esquema completo de configuración está aquí. Remarcaremos no-exhaustivamente las partes importantes en prosa aquí.

name, namespace

name es el nombre de la librería, usado como su "área de nombre". Puedes activar el área de nombre mediante "namespace": true en tu bsconfig.json. El área de nombre es casi obligatoria; no lo hemos activado por defecto aún para preservar la compatibilidad hacia atrás.

Explicación: por defecto, tus archivos, una vez usados como una dependencia de terceros, están disponibles globalmente al consumidor. Ejemplo, si tienes un Util.re y el consumidor también tiene un archivo del mismo nombre, estos chocarán. Activar namespace evita esto envolviendo todos los archivos de tu proyecto en una capa de módulo extra; en vez de un módulo global Util, el consumidor verá el tuyo como MyProject.Util. El nombrado afecta a tus consumidores, no a ti mismo.

También es conocido qué, en Bsb, "namespace" es sólo un término elegante para un modulo autogenerado que envuelve todos los archivos de tu proyecto (eficiente y correctamente ¡por supuesto!) para un consumo de terceros.

No hacemos nombrado a nivel de carpetas para tu propio proyecto; todos tus archivos deben tener nombres únicos. Esta es una limitación que permite muchas características como búsqueda rápida y una reorganización de proyectos más sencilla.

Nota: el name de bsconfig.json debería ser el mismo name de package.json, para evitar casos de esquina confusos. Sin embargo, esto significa que no puedes usar nombres camelCased como MyProject, ya que package.json y npm te prohíben hacerlo (algunos sistemas de archivo distinguen entre mayúsculas y minúsculas). Para tener el "área de nombre"/módulo como MyProject, escribe "name": "my-project". Bsb convertirá eso en un nombre camelCased correctamente.

orígenes

Los archivos de origen necesitan ser especificados explícitamente (no queremos accidentalmente perforar en algunos directorios que no estén relacionados). Ejemplos:

{
  "fuentes": ["src", "ejemplos"]
}

{
  "fuentes": {
    "dir": "src",
    "subdirs": ["página"]
  }
}

{
  "fuentes": [
    "ejemplos",
    {
      "dir": "src",
      "subdirs": true // recursivamente construye cada directorio
    }
  ]
}

Puedes marcar tus directorios como "sólo desarrollo" (por ejemplo, para pruebas). Estos no serán construidos y expuestos para terceros, o incluso para otros directorios de desarrollo en el mismo proyecto:

{
  "sources" : {
    "dir" : "test",
    "type" : "dev"
  }
}

dependencias-bs, depedencias-bs-dev

Lista de dependencias BuckleScript/Reason. Así como las dependencias de package.json, serán buscadas en node_modules.

reason, refmt

La configuración reason es activada por defecto. Para activar JSX para ReasonReact, especifica:

{
  "reason": {"react-jsx": 2}
}

La configuración refmt debería ser especificada explícitamente: ·refmt": 3 para la nueva sintaxis de Reason V3.

js-post-build

Un gancho que se invoca cada vez que un archivo se vuelve a compilar. Bueno para la interoperabilidad de sistemas basados en JS, pero por favor, úsalo con moderación. Llamando tu comando personalizado para cada archivo recompilado demora tu construcción y empeora la experiencia de construcción incluso para terceros que usan tu librería.

Ejemplo:

{
  "js-post-build": {
    "cmd": "/path/to/node ../../postProcessTheFile.js"
  }
}

Ten en cuenta que la resolución de la ruta de comandos se hace mediante lo siguiente:

  • /myCommand se resuelve en /myCommand
  • myCommand/ se resuelve en node_modules/myCommand
  • ./myCommand se resuelve en myProjectRoot/myCommand
  • myCommand sólo es llamado como myCommand. Pero nota que Bsb no lee en tu ambiente, así que si pones node, no lo encontrará a menos que especifiques un camino absoluto. Alternativamente, apunta a un script que tenga el encabezado #!/usr/local/bin/node.

El mismo comando es llamado desde dentro de lib/bs.

package-specs

La salida de bien CommonJS (por defecto), módulos ES6 o AMD. Ejemplo:

{
  "package-specs": {
    "module": "commonjs",
    "in-source": true
  }
}
  • "module": "es6-global" resuelve node_modules usando caminos relativos. Bueno para el uso de desarrollo-tiempo para una conjunción ES6 con navegadores como Safari y Firefox que son compatibles con módulos ES6 hoy. ¡No más empaquetamiento desarrollo-tiempo!
  • "in-source": true genera una salida junto a los archivos de origen. Si lo omites, generará artefacto en lib/js. El directorio de salida no es configurable de otra manera.

Esta configuración sólo aplica para ti, cuando desarrollas el proyecto. Cuando el proyecto es usado como una librería de terceros, los bsconfig.json package-specs del mismo consumidor sobreescriben la configuración aquí, logicamente.

sufijo

Pueden ser ".js" o ".bs.js". Recomendamos fuertemente el último.

Decisiones de Diseño

Generando archivos JS con el sufijo .bs.js significa que, en el lado de JS, puedes hacer const myBuckleScriptFile = require('./theFile.bs'). Los beneficios:

  • Es inmediatamente claro que estamos trabajando con un archivo JS generado aquí.
  • Esto evita choques con un potencial archivo theFile.js en el mismo archivo.
  • Esto evita la necesidad de usar un cargador de sistema de construcción para archivos BuckleScript. Este construcción en origen significa integrar un proyecto BuckleScript en tu código base JS básicamente no toca nada en tu tubería de construcción.
  • El sufijo .bs.js deja que bsb siga artefactos JS mucho mejor.

advertencias

Selectivamente enciende o apaga ciertas advertencias y/o conviértelas en errores fuertes. Por ejemplo:

{
  "warnings": {
    "number": "-44-102",
    "error": "+5"
  }
}

Desactiva la advertencia 44 y 102 (comparación polimórfica). Convierte la advertencia 5 (aplicación parcial cuyo resultado tiene un tipo de función y es ignorado) en un error fuerte.

El número de advertencia se muestra en la salida de construcción cuando son activados. La lista completa está aquí, un poco más abajo. 100 y mayores son específicos de BuckleScript.

bsc-flags

Bandera extra para pasar a la llamada bsc escondida. Notablemente: ["-bs-super-errors"] para activar el limpiador de errores de salida. No hay necesidad de pasar este indicador para un proyecto de Reason; está activado por defecto.
