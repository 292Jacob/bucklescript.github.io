Insertar JavaScript crudo

Estamos introduciendo esta trampa de escape de último recurso primero, en caso de que te atores intentando las otras APIs más legítimas y quieras seguir adelante. Aquí está como puedes soltar un trozo de JavaScript directo en tu archivo BuckleScript:

let add = [%raw {|
  function(a, b) {
    console.log('¡hola desde JavaScript crudo!');
    return a + b;
  }
|}]

let _ = Js.log (add 1 2)

Sintaxis de Reason:

let add = [%raw {|
  function(a, b) {
    console.log("¡hola desde JavaScript crudo!");
    return a + b
  }
|}];

Js.log(add(1, 2));

La sintaxis {|foo|} significa para las multi-líneas de OCaml/BuckleScript/Reason, una sintaxis de "cadena citada". Piensa en ellos como el equivalente de plantillas literales de JavaScript. Ningún escape es necesario dentro de esa cadena.

Cuidado con la sintaxis de OCaml/Reason aquí. [%raw foo] te permite incrustar una expresión. Para declaraciones de alto nivel en OCaml/Reason, usa [%%raw foo] (dos %):

[%%raw "var a = 1"]

let f = [%raw "function() {return 1}"]

Sintaxis de Reason:

[%%raw "var a = 1"];

let f = [%raw "function() {return 1}"];

You can also pass a function declaration with a string body in raw:

let f: (int -> int -> int) = [%raw fun a b -> "{return a + b}"]

Reason syntax:

let f: (int, int) => int = [%raw (a, b) => "{return a + b}"];

Output:

var f = function (a,b){{return a + b}};

If you have to use raw in the first place, then prefer this format when you're passing a function. This allows the compiler to understand that it's a function, to see the number of arguments it has, to detect some side-effects and to generate better JS code. Also note that while the function's type annotation ((int -> int -> int) in the preceding OCaml example) is not strictly required, it gives the compiler a better chance to understand your raw JavaScript, and you may get a compiler warning without it.

Depurador

You can also drop a [%debugger] expression in a body:

let f x y =
  [%debugger];
  x + y

Reason syntax:

let f = (x, y) => {
  [%debugger];
  x + y
};

Output:

function f (x,y) {
  debugger; // JavaScript developer tools will set an breakpoint and stop here
  x + y;
}

Detectar Variables Globales

BuckleScript provides a relatively type safe approach for such use case: external. [%external a_single_identifier] is a value of type option. Example:

match [%external __DEV__] with
| Some _ -> Js.log "dev mode"
| None -> Js.log "production mode"
Reason syntax 
switch ([%external __DEV__]) {
| Some(_) => Js.log("dev mode")
| None => Js.log("production mode")
};

Output:

var match = typeof (__DEV__) === "undefined" ? undefined : (__DEV__);

if (match !== undefined) {
  console.log("dev mode");
} else {
  console.log("production mode");
}

Another example:

match [%external __filename] with
| Some f -> Js.log f
| None -> Js.log "non-node environment"

Reason syntax

switch ([%external __filename]) {
| Some(f) => Js.log(f)
| None => Js.log("non-node environment")
};

Output:

var match = typeof (__filename) === "undefined" ? undefined : (__filename);

if (match !== undefined) {
  console.log(match);
} else {
  console.log("non-node environment");
}

Consejos & Sugerencias

Embedding raw JS snippets is highly discouraged, though also highly useful if you're just starting out. As a matter of fact, the first few Reason BuckleScript projects were converted through:

  • pegar trozos JS crudos dentro de un archivo
  • examinar la salida de JS (idéntica)
  • gradualmente extraer unos pocos valores y funciones y asegurándose que la salida aún se viera bien

At the end, we get a fully safe, converted Reason BuckleScript file whose JS output is clean enough that we can confidently assert that no new bug has been introduced during the conversion process.

We have a small guide on this iteration here. Feel free to peruse it later.
