Generar convertidores y ayudantes

Algunas veces, quizás quieras generar por ejemplo, funciones accesoras de una declaración de variante, o un objeto Js.t + funciones convertidoras de una definición de registro. BuckleScript viene con unas pequeñas notaciones que te permiten generarlos.

Funciones & Valores Planos para Variantes

Usa accessors.

type action =
  | Click
  | Envío de cadena
  | Cancelar
  [@@bs.deriving accessors]

Sintaxis de Reason:

[@bs.deriving accessors]
type action =
  | Click
  | Envío(cadena)
  | Cancelar;

Constructores de variantes con cargas generan funciones, constructores sin carga generan enteros planos. Nota:

  • Los accesores generados son minúsculas.
  • ¡Ahora puedes usar estos ayudantes en el lado JavaScript! Pero no dependas de sus valores reales, por favor.
  • No funciona con variantes polimórficas aún.

Salida:

function submit(param_0) {
  return /* Enviar */[param_0];
}

var click = /* Click */0;

var cancel = /* Cancelar */1;

exports.click  = click;
exports.submit = submit;
exports.cancel = cancel;

Uso

let s = submit "hola" (* da Enviar "hola" *)

Sintaxis de Reason:

let s = submit("hola"); /* da enviar("hola") */

Esto es útil:

  • Cuando estás pasando una función accesora como una función de alto nivel (cosa que no son los constructores de variantes planas).
  • Cuando te gustaría que el lado JS use estos valores & funciones opacamente y te devuelvan un constructor de variante (ya que JS no tiene algo así).

Convertir Entre Objeto y Registro de Js.t

Usa jsConverter.

type coordinates = {
  x : int;
  y : int
} [@@bs.deriving jsConverter]

Sintaxis de Reason:

[@bs.deriving jsConverter]
type coordinates = {
  x: int,
  y: int
};

Genera 2 funciones del siguiente tipo:

val coordinatesToJs : coordinates -> <x : int ; y : int > Js.t

val coordinatesFromJs : < x: int ; y : int ; .. > Js.t -> coordinates

Sintaxis de Reason:

let coordinatesToJs: coordinates => {. "x": int, "y": int};

let coordinatesFromJs: {.. "x": int, "y": int} => coordinates;

Nota:

  • coordinatesFromJs usa un tipo de objeto abierto que acepta más campos, sólo para ser más permisivo.
  • Los convertidores son superficiales. No penetran recursivamente en los campos y los convierten. Esto preserva la velocidad y simplicidad de la salida mientras satisfacen el 80% de los casos. ¡Envía un problema si te gustaría ver la conversión recursiva!

Uso

Esto exporta un objeto JS jsCoordinates (¡no un registro!) para que lo usen los archivos JS:

let jsCoordinates = coordinatesToJs {x = 1; y = 2}

Sintaxis de Reason:

let jsCoordinates = coordinatesToJs({x: 1, y: 2});

Esto enlaza a un registro jsCoordinates (¡no un objeto!) que existe en el lado JS, presuntamente creado por JS llamando a la función coordinatesFromJs:

external jsCoordinates: coordinates = "jsCoordinates" [@@bs.module "myGame"]

Sintaxis de Reason:

[@bs.module "myGame"] external jsCoordinates : coordinates = "jsCoordinates";

Más Seguridad

La función generada anteriormente usa objetos del tipo Js.t. Puedes además esconder este detalle de implementación haciendo el tipo de objeto abstracto mediante newType:

type coordinates = {
  x : int;
  y : int
} [@@bs.deriving {jsConverter = newType}]

Sintaxis de Reason:

[@bs.deriving {jsConverter: newType}]
type coordinates = {
  x: int,
  y: int
};

Genera dos funciones del siguiente tipo:

val coordinatesToJs : coordinates -> abs_coordinates

val coordinatesFromJs : abs_coordinates -> coordinates

Sintaxis de Reason:

let coordinatesToJs: coordinates => abs_coordinates;

let coordinatesFromJs: abs_coordinates => coordinates;

Uso

Usando newType, has prevenido a los consumidores de inadvertidamente hacer lo siguiente:

let myCoordinates = {
  x = 10;
  y = 20
}
let jsCoords = coordinatesToJs myCoordinates

let x = jsCoords##x (* ¡No permitido! No accedas a los detalles internos del objeto *)

Sintaxis de Reason:

let myCoordinates = {
  x: 10,
  y: 20
};
let jsCoords = coordinatesToJs(myCoordinates);

let x = jsCoords##x; /* ¡No permitido! No accedas a los detalles internos del objeto */

La misma salida generada. ¿No es genial que los tipos previenen accesos inválidos que de otra manera tendrías que codificar en el tiempo de ejecución?

Convertir entre Enteros JS Enumerados y Variantes BS

Usa jsConverter.

type fruit =
  | Manzana
  | Naranja
  | Kiwi
  | Sandía
  [@@bs.deriving jsConverter]

Sintaxis de Reason:

[@bs.deriving jsConverter]
type fruit =
  | Manzana
  | Naranja
  | Kiwi
  | Sandía;

Genera 2 funciones de los siguientes tipos:

val fruitToJs : fruit -> int

val fruitFromJs : int -> fruit option

Sintaxis de Reason:

let fruitToJs: fruit => int;

let fruitFromJs: int => option(fruit);

Para fruitToJs, cada constructor de variante de fruta mapearía a un entero, empezando en 0, en el orden en el que son declarados.

Para fruitFromJs, el valor de retorno es un option, porque no todos los int mapean a un constructor.

Además puedes adjuntar un [@bs.as alternativeIntValue] a cada constructor para personalizar su salida.

Uso

type fruit =
  | Manzana
  | Naranja [@bs.as 10]
  | Kiwi [@bs.as 100]
  | Sandía
  [@@bs.deriving jsConverter]

let zero = fruitToJs Apple (* 0 *)

let _ = match fruitFromJs 100 with
| Some Kiwi -> Js.log "this is Kiwi"
| _ -> Js.log "se ha recibido algo mal del lado JS"

Sintaxis de Reason:

[@bs.deriving jsConverter]
type fruit =
  | Manzana
  | [@bs.as 10] Naranja
  | [@bs.as 100] Kiwi
  | Sandía;

let zero = fruitToJs(Manzana); /* 0 */

switch (fruitFromJs(100)) {
| Some(Kiwi) => Js.log("esto es Kiwi")
| _ => Js.log("se ha recibido algo mal del lado JS")
};

Nota: usando bs.as aquí, todos los números de codificación subsecuentes cambian. ¡Apple sigue siendo ``, Orange es 10, Kiwi es 100 y Watermelon es 101!

Más Seguridad

Similar al objeto JS <-> derivado de un registro, puedes esconder el hecho de que JS enum son ints usando newType:

type fruit =
  | Manzana
  | Kiwi [@bs.as 100]
  | Sandía
  [@@bs.deriving {jsConverter = newType}]

Sintaxis de Reason:

[@bs.deriving {jsConverter: newType}]
type fruit =
  | Manzana
  | [@bs.as 100] Kiwi
  | Sandía;

Genera 2 funciones del siguiente tipo:

val fruitToJs : fruit -> abs_fruit

val fruitFromJs : abs_fruit -> fruit

Sintaxis de Reason:

let fruitToJs: fruit => abs_fruit;

let fruitFromJs: abs_fruit => fruit;

Para fruitFromJs, el valor de retorno, a diferencia de los anteriores casos de tipos no abstractos, no contiene un option, porque no hay manera de pasar un valor erróneo; ¡el único creador de valores abs_fruit es fruitToJs!

Uso

type fruit =
  | Manzana
  | Kiwi [@bs.as 100]
  | Sandía
  [@@bs.deriving {jsConverter = newType}]

let opaqueValue = fruitToJs Manzana

external jsKiwi: abs_fruit = "JuroQueEstoEsUnKiwi" [@@bs.module "myJSFruits"]
let kiwi = fruitFromJs jsKiwi

let error = fruitFromJs 100 (* no, no se puede tomar un int al azar *)

Sintaxis de Reason:

[@bs.deriving {jsConverter: newType}]
type fruit =
  | Manzana
  | [@bs.as 100] Kiwi
  | Sandía;

let opaqueValue = fruitToJs(Manzana);

[@bs.module "myJSFruits"] external jsKiwi : abs_fruit = "JuroQueEstoEsUnKiwi";
let kiwi = fruitFromJs(jsKiwi);

let error = fruitFromJs(100); /* no, no se puede tomar un int al azar*/

Convertir entre Cadenas Enum JS y Variantes BS Polimórficas

Similar a la sección previa, excepto que las variantes polimórficas son convertidas en string en vez de int.

Uso

type fruit = [
  | `Manzana
  | `Kiwi [@bs.as "miniCoco"]
  | `Sandía
] [@@bs.deriving jsConverter]

let appleString = fruitToJs `Manzana (* Manzana" *)
let kiwiString = fruitToJs `Kiwi (* "miniCoco" *)

Sintaxis de Reason:

[@bs.deriving jsConverter]
type fruit = [
  | `Manzana
  | [@bs.as "miniCoco"] `Kiwi
  | `Sandía
];

let appleString = fruitToJs(`Manzana); /* "Manzana" */
let kiwiString = fruitToJs(`Kiwi); /* "miniCoco" */

Derivar convertidores con tipos abstractos mediante newType aún funciona.

Abstract Type

(Since 3.0.0) for platform agnostic code and fine tuning of runtime representation.

OCaml has multiple backends. For the record type, it has quite similar representations between native and JS backend: an array of values.

Sometimes, we want to have optimal runtime encoding while still make sure the code portable: bs.deriving can accept abstract for record types to completely hide the record but only expose its getter/setter/maker. This is helpful for writing cross platform code.

In ml files

type cord = {
  x : int;
  mutable y : int;
} [@@bs.deriving abstract]

It will generate such functions

type cord 
val cord : a:int -> b:int -> t  (* maker *)
val x : cord -> int (* getter *)
val y : cord -> int (* getter *)
val ySet : cord -> int -> unit (* setter *)

Since now cord is completey abstract type, for BuckleScript backend, we compile it into a js object instead of array:

let u = cord ~x:2 ~y:3

let () = 
  let x, y = u |. x, u |. y in 
  u |. ySet (x + y)

Will generate such code

var u = {
  x: 2,
  y: 3
};

var x = u.x;

var y = u.y;

u.y = x + y | 0;

bs.as to customize fields

We can customize the label to be different names, suppose we change the type definition as below:

type cord = {
  x : int;
  mutable y : int [@bs.as "Content-Type"]; 
} [@@bs.deriving abstract]


let u = cord ~x:2 ~y:3

let () = 
  let x, y = u |. x, u |. y in 
  u |. ySet (x + y)

It will generate code as below:

var u = {
  x: 2,
  "Content-Type": 3
};

var x = u.x;

var y = u["Content-Type"];

u["Content-Type"] = x + y | 0;

bs.optional for the record field

We can also mark some labels optional, so that it does not need to be supplied at the initialization time:

type cord = {
  mutable x : int [@bs.optional]; 
  y : int ; 
} [@@bs.deriving abstract]

It will generate such functions:

type cord
val cord : ?x:int -> y:int -> unit -> cord 
val x : cord -> int option 
val xSet : cord -> int -> unit 
val y : cord -> int 

let u = cord  ~y:3 ()

let () = 
  let x, y = u |. x, u |. y in 
  match x with 
  | None -> 
    u |. xSet y
  | Some x -> 
    u |. xSet (x + y )

The generated output is:

var u = {
  y: 3
};

var x = u.x;

var y = u.y;

if (x !== undefined) {
  u.x = x + y | 0;
} else {
  u.x = y;
}

fine access control in signatures

bs.deriving abstract is also applicable in mli files:

type cord = {
  mutable x : int  [@bs.optional]; 
  y : int ; 
} [@@bs.deriving abstract]

Will generate such signatures

val cord : ?x:int -> y:int -> unit -> cord
val x : cord -> int option
val xSet : cord -> int -> unit 
val y : cord -> int

We can make it a bit more restrictive when exposing to clients by removing mutable in x:

type cord = {
  x : int [@bs.optional];
  y : int
} [@@bs.deriving abstract]

Will generate such signatures:

val cord : ?x:int -> y: int -> unit -> cord  
val x : cord -> int option
val y : cord -> int

More restrictive by marking the contructor private:

type cord = private { x : int [@bs.optional]; y : int} [@@bs.deriving abstract]

Will generate such signatures:

val x : cord -> int option
val y : cord -> int
