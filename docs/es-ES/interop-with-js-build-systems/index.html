Interop with JS Build System

Si vienes de JS, lo más probable es que ya tengas un sistema de construcción en tu proyecto existente. Aquí hay un resumen de todo el papel que jugaría bsb en tu línea de construcción, si quieres introducir algunos códigos BS en tu base de código.

Sistemas de Construcción JS Populares

El ecosistema JS usa unos pocos sistemas de construcción: browserify, rollup, webpack, etc. El último es posiblemente el más popular de estos tres (para Diciembre del 2017 =P). Estos sistemas de construcción hacen tanto la compilación como el enlace (es decir, empaquetan muchos archivos en uno o varios archivos).

BuckleScript y bsb sólo se ocupan del paso de compilación; mapea un archivo ml/re/mli/rei en un archivo JS de salida. Así, en teoría, ninguna integración del sistema de construcción es necesario desde nuestro lado. Desde, por ejemplo, la perspectiva del vigía de webpack, los archivos JS de BuckleScript generados son casi equivalentes a tus archivos JS escritos a mano. Además recomendamos que inicialmente verifiques esos archivos JS generados por BS, como dice este flujo de trabajo:

  • Puedes introducir BuckleScript y/o Reason silenciosamente en tu base de código sin molestar una infraestructura de construcción existente.
  • Tienes una diferencia visual en el rendimiento & corrección de tu archivo JS cuando actualizas los archivos ml/re y los artefactos cambian.
  • Puedes dejar que tus compañeros parcheen en caliente los archivos JS en situaciones de emergencia sin que tengan que aprender BS/Reason.
  • Puedes eliminar BuckleScript y Reason completamente desde tu base de códigos y las cosas aún funcionarían (en caso de que tu compañía decida dejar de usarnos).

La pequeña desventaja de estos enfoques es que necesitas tener tanto un vigía de bsb como un vigía de webpack abiertos. Pero si activas la característica bsb de VSCode-Reasonml, evitas lo anterior casi que sin problemas.

Para lo que vale, también puedes volver bsb un paso automatizado en tu línea de constucción (p.e. en un cargador webpack, como nuestro bs-loader en desuso aquí).

Consejos & Sugerencias

Puedes hacer que los archivos BS JS parezcan aún más idiomáticos mediante la configuración del sufijo de origen en bsconfig.json:

{
  "package-specs": {
    "module": "commonjs", // o cualquier sistema de módulos que tu proyecto use
    "in-source": true
  },
  "suffix": ".bs.js"
}

Esto:

  • Generará los archivos JS junto a tus archivos de origen BS.
  • Usará la extensión de archivo .bs.js, así que puedes requerir estos archivos en el lado js mediante require('./MyFile.bs'), sin necesidad de un cargador.

Usar Cargadores en el Lado BS

"¿Qué pasa si mi sistema de construcción usa un CSS/png/loquesea y me gustaría usarlo en BuckleScript?"

Loaders are indeed troublesome; in the meantime, please use e.g. [%raw {|require('./myStyles.css')|}] at the top of your file. This just uses raw to compile the snippet into an actual JS require.
