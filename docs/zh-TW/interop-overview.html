概覽

"Interop"（"interoperability" 的縮寫），BuckleScript 的情境中，表示與 JavaScript 的溝通。 BuckleScript 其中一個主要目標就是盡可能與現有 JS 堆疊流利地交互作用。

一個真實，具規模的程式庫，無法簡單地在幾天，幾週或甚至幾個月或年內將某個技術轉換成另一個。 若沒有經過仔細設計交互作用便將語言帶入程式庫，轉換成本便會使得語言帶來的好處降低進而成為門檻。

精心製作簡單，高效，本地及非規定性的交互作用系統，讓開發者在嘗試這個科技過程中使用熟悉的概念，對於未知性能夠給予足夠的信心推薦給主管，並且在轉換過程中保持高生產率。

實作

上述章節可能聽起來有點時髦術語；這裡是我們如何設計 BuckleScript 的交互作用，從堆疊最低的部分到最高。

變數

大多數的變數名稱編譯成乾淨的 JS 名稱。 hello 編譯成 hello，而其他編譯器可能編譯成 __$INTERNAL$hello$1。 乾淨名稱（例如 "無名稱重整"）有助於除錯及一次性實驗。 我們曾看過團隊某位成員不熟悉 BuckleScript 及 Reason，直接在 BuckleScript 程式碼中投入一段原生 JS 程式碼區塊做的緊急修復。 知道最後逃生出口感到欣慰。

資料結構

主要在 BuckleScript 內的資料結狗乾淨地對應到 JS。例如，BS 字串就是 JS 字串。BS 陣列就是 JS 陣列。下列 BS 程式碼：

let messages = [| "hello"; "world"; "how"; "are"; "you" |]

let messages = [|"hello", "world", "how", "are", "you"|];

粗略地編譯成 JS 程式碼：

var messages = ["hello", "world", "how", "are", "you"]

使用這些值的時候無須腦力及效能負荷。自然地，BuckleScript 方面的值自動成為字串陣列型別。

這個行為不適用於複雜資料結構；每個部份都有專屬章節說明。

函數

多數情況下，你可以直接在 BS 呼叫 JS 函數，反之亦然！大多數都不會有效能負荷。

模組/文件

BS 中預設每個 let 宣告都會輸出並且可以從 JS 使用。 另一方面，可以在 BS 檔案中宣告使用哪些 JS 模組。 可以輸出或匯入 CommonJS，ES6 及 AMD 模組。

產生的乾淨結果足以超越（縮排有點差）手工撰寫的 JavaScript 程式碼。試試在遊樂產執行幾個程式碼片段！

建置系統

BS 內建快速的（最快？）建置系統。 啟動及結束於幾毫秒時間，非常快速。 增量編譯於兩位數毫秒時間。 讓建置系統能夠隱形地加入到整個 JS 建置管線而不會感到難堪。 除非你的 JS 建置管線已經是慢到尷尬。 沒關係。

1 個 BS 檔案編譯出 1 個 JS 檔案。 建置可以設定成將產生出來的 JS 檔案放在 BS 檔案旁邊或者不在 ml/re 檔案旁邊。 This means you don't have to ask the infra team's help in trying out BuckleScript at your company; simply generate the JS files and check them in. 從其他編譯管線的角度來看，就像是你自己撰寫這些 JS 檔案。 這就是 Messenger 如何成功地將 Reason 導入其程式庫。

套件管理

使用 NPM 或 Yarn。 由於產生的結果足夠整潔，你可以在 NPM prepublishOnly 時期發佈並刪除預先所有 BuckleScript 足跡。 使用者甚至不知道你是使用 BS 撰寫而非 JS！ 立即檢查 node_modules；可能會找到一些正在使用的 BS 程式碼而不知覺！
